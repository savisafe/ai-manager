# План развития (черновик)

Дорожная карта по фазам. Параллельно можно вести работы там, где нет жёсткой зависимости. Текущий статус реализации — в `DEVELOPMENT_CONTEXT.md`.

## Фаза A — Надёжность и продакшен-поток

- Воркер BullMQ + Redis: асинхронная обработка после вебхука, лимиты параллелизма, повтор при падении воркера.
- Retry/backoff на отправку в Telegram/WhatsApp API; явные коды ошибок в логах.
- Production URL вебхуков, секреты и проверка подписей (WhatsApp уже есть).
- Опционально: Sentry или аналог для необработанных исключений.

## Фаза B — Продукт и скрипты

- Доработка sales-FSM: отдельные ветки возражений (как в `PROMPT_PLAN.md` §6.3), явный шаг сбора контактов при необходимости.
- A/B варианты реплик и CTA в конфиге; привязка к `conversation` или эксперименту.
- Уведомление менеджера при `handoff` (Telegram-группа, email, внутренний webhook).

## Фаза C — Каналы и архитектура

- Общий интерфейс канала (`ChannelAdapter`): единая модель входящего сообщения и отправки ответа (рефакторинг из двух модулей).
- Валидация `sales-scripts.json` и профилей промпта через Zod (или аналог) при старте приложения.

## Фаза D — LLM и качество

- Абстракция `LLMProvider`: Ollama / OpenAI / др. через конфиг, единый контракт и таймауты.
- Guardrails: пост-валидация ответа (длина, запрещённые темы), при необходимости повтор или fallback на скрипт.
- Метрики: время ответа, переходы по стадиям, handoff rate; простой экспорт или таблица агрегатов.

## Фаза E — Масштаб и CRM (после стабилизации MVP)

- Кэш сессий/частых ответов в Redis при росте нагрузки.
- Модуль CRM/лидов: статусы, воронка, экспорт; при необходимости отдельное приложение или интеграция.
